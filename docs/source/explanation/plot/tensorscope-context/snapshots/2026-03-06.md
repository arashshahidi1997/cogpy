# TensorScope Context Snapshot — 2026-03-06

This snapshot captures current working context and root-cause findings relevant to PSD Explorer + core views.

## Goal

- Stabilize PSD Explorer UX and remove duplicated functionality by relying on shared TensorScope state (time, selection, processing).
- Identify where specific UI/plot behaviors stem from (autoscaling, overview coupling, processing application in spatial map).

## Observed Behavior

- Switching PSD method (Welch → multitaper) can cause the **average PSD** to appear “flat / invisible” until zooming.
- When scroll-zooming the **timeseries Y axis**, the **overview strip** can also change scale.
- Ambiguity: whether the “instantaneous” **spatial LFP map** shows the raw signal or the processed (processing-chain) signal.

## Suspected Root Cause (with code pointers)

### 1) Welch → multitaper: average PSD scale not readapting

Hypothesis:
- Autoscaling is not being applied to the final element returned by the average-PSD `DynamicMap`.
- When composing `curve * area * indicator`, some options can be lost if applied only to the base curve.

Where to inspect:
- Average PSD construction inside the PSD explorer layer/module.

Search recipe:
```bash

rg -n "psd_explorer" code/lib/cogpy/src/cogpy/core/plot/tensorscope -S
```

### 2) Timeseries overview: what it is + why Y zoom can couple

Overview meaning:
- The overview strip is a **summary trace** used for navigation (commonly mean across channels) and is typically downsampled for speed.

How Y coupling can happen:
- shared Bokeh `Range1d` models or shared HoloViews axes in a composed layout
- linking utilities that accidentally propagate Y range updates (even when only X linking is intended)

Where to inspect:
- multichannel viewer overview creation (summary signal definition + axis options)
- any link between overview and detail plots (often only X is intended)

Search recipe:
```bash
rg -n "Overview|mean\\(|RangeToolLink|shared_axes|Range1d|ylim" code/lib/cogpy/src/cogpy/core/plot -S
```

### 3) Spatial LFP: processed vs raw (instantaneous mode)

Hypothesis:
- The instantaneous spatial view may index directly into the raw tensor at `t_selected`.
- The processing chain is often applied through “window extraction” helpers; instantaneous mode may bypass those helpers and therefore show raw data.

Where to inspect:
- the spatial map layer implementation
- any underlying “frame element” / “grid frame” helper that computes a 2D scalar field
- how `ProcessingChain` is applied (windowed vs direct)

Search recipe:
```bash
rg -n "class Spatial|spatial_map|instant|rms|get_window|ProcessingChain" code/lib/cogpy/src/cogpy/core/plot -S
```

## Confirmed Root Cause (current best understanding)

### Welch → multitaper scale
- Root cause mechanism: **autoscale settings can be dropped** when returning a composed/overlaid HoloViews element unless applied to the final composed object.

### Timeseries overview coupling
- Root cause mechanism: coupling is typically due to **shared range models** or a linking utility that is broader than intended. The first place to verify is whether the overview and detail plots share any Bokeh range objects, and whether only X is linked.

### Spatial LFP processed vs raw
- Root cause mechanism: processing can be applied only when using processing-chain window helpers; any path that indexes “instantaneous frame” directly from the raw tensor will display raw values.

## Minimal Fix Plan + Acceptance Checks

### Fix plan (mechanism-level)
- Average PSD autoscaling: ensure `framewise=True` (or equivalent range behavior) is applied to the **final** returned element after overlays are composed.
- Timeseries overview: ensure overview Y range is independent and not linked to detail Y range; linking should remain **X-only**.
- Spatial instantaneous map: route instantaneous frame computation through a processing-aware path (or explicitly document it as raw-only).

### Acceptance checks
- Switching PSD method changes the visible power range without manual zooming.
- Zooming Y on timeseries does not alter overview Y scaling.
- Spatial instantaneous map clearly matches processed output when processing is enabled (or the UI clearly states it is raw).

## What Changed in Understanding

- PSD Explorer is best treated as *PSD-specific additions* (transform + views) on top of existing TensorScope state/views.
- Many UI issues are not “PSD math bugs” but **plot/model linking** issues (autoscale + shared ranges).
