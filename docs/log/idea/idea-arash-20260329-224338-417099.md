Feature request: @api_contract decorator for cogpy functions

## Problem

Agents generating code with cogpy write unnecessary for-loops because they don't know which functions vectorize over xarray dims. Example: looping over (AP, ML) channels calling psd_multitaper per-channel, when psd_multitaper already handles multi-dim xarray natively.

## Proposed solution

Add an `@api_contract` decorator to cogpy's most-used functions that declares:
- Input dim contracts (which dims are required, which are broadcast over)
- Output dim contracts (what dims the result has)
- Vectorization behavior

```python
from cogpy.datasets.contracts import api_contract

@api_contract(
    input={"signal": "DataArray[..., time]"},
    output={"psd": "DataArray[..., freq]", "freqs": "ndarray[freq]"},
    vectorizes_over="all non-time dims",
)
def psd_multitaper(signal, fs, NW=4.0, ...):
    ...
```

The decorator:
- Is introspectable by external tools (codio_ingest_api, MCP)
- Serves as documentation for human readers
- Could optionally validate inputs at runtime (debug mode)
- Builds on existing schemas.py DIMS_* conventions

## Priority functions to annotate

Start with the functions most commonly used in notebooks:
- cogpy.spectral.psd: psd_multitaper, psd_welch
- cogpy.spectral.specx: psdx, spectrogramx
- cogpy.spectral.features: band_power, narrowband_ratio, ftest_line_scan
- cogpy.measures.spatial: moran_i, gradient_anisotropy
- cogpy.brainstates.intervals: perievent_epochs
- cogpy.triggered: triggered_average, triggered_std, estimate_template

## Relationship to schemas.py

schemas.py defines DIMS_* constants and validate_*/coerce_* functions for data containers. @api_contract extends this to function-level contracts — "this function takes DIMS_SIGNAL and returns DIMS_PSD."
